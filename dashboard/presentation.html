The Challenge

Manual QA doesn't scale

As NETGEAR Insight evolves — new subscription tiers, Stripe migrations, multi-country support — the QA burden grows exponentially. Without automation, every release is a risk.

⏳

Hours of Manual Effort

Each release cycle requires manually stepping through subscription flows across multiple accounts, countries, and plan types — taking QA engineers hours per drop.

🔍

Zero Visibility During Execution

No live view of what's passing, failing, or still running. Results only known at the very end — too late to intervene when something goes wrong mid-suite.

📂

No Evidence Trail

Manual testing leaves no systematic proof. Screenshots are ad-hoc, steps undocumented, and reproducing a failure requires the same manual work all over again.

🌍

Multi-Country Complexity

Stripe pricing varies across 38+ countries with different currencies, tax rules, and checkout flows. Testing even a subset manually is impractical per release.

The Solution

A closed-loop QA platform, built in-house

The QA Automation Dashboard connects Playwright test execution to a real-time WebSocket UI — giving engineers live feedback, automatic history, and evidence-grade screenshots on every single run.

Dashboard Deep Dive

One platform. Total visibility.

Every tab solves a distinct QA problem — from live test monitoring to multi-country Stripe validation. Together they form the complete picture of a release cycle.

1
Current Run

Live Test Execution Monitor

The heartbeat of the dashboard. Every test case is streamed live via WebSocket — you watch each TC flip from Pending → Running → Passed / Failed the instant the Playwright reporter writes the update. Step-level detail, error messages, and evidence screenshots are all expandable inline.

The suite selector lets you choose between the full 5-TC hackathon suite, the 2-TC quick demo, the Drop5 production suite, or individual plan suites — then click ▶ Run Tests to launch instantly.

WebSocket real-time Step-level detail Evidence screenshots Progress bar Suite selector Kill All control
2
Run History

Persistent Audit Trail of Every Run

Every time a test suite completes, the result is automatically serialised to dashboard/history/run-<timestamp>.json and surfaced in this tab. The last 20 runs are indexed — expanding any row shows the full per-test breakdown, identical to the Current Run view.

For compliance and retrospectives: you always know which tests passed on which day, what errors they hit, and how long each run took. No extra tooling required.

Auto-saved on completion 20-run index Expandable history detail Pass/fail trend Timestamps + duration
3
DirectPro Purchases

Business Data Tracking for Drop 5 Migration

During the Drop 5 release, NETGEAR needed to track credit purchase status across dozens of international accounts. This tab reads live from im10-dpro-accounts.csv and presents a filterable progress board — how many accounts are Purchased, In Queue, Skipped (CA), or Failed.

The stacked progress bar gives an at-a-glance view of migration completeness. Filter buttons isolate any status for focused attention.

CSV-driven live data Status filters Stacked progress bar Credit totals Team accounts
4
Account Navigator

Automated Account Lifecycle Management

The Account Navigator is a full account management tool embedded directly in the dashboard — no separate tab, no extra port. It orchestrates the complete QA lifecycle: account creation, org/location setup, device onboarding, subscription purchase, and Stripe checkout — all via API calls that drive a headless Playwright session.

The Mongo DB state panel (Grace / Expired / Active) lets QA engineers advance subscription states without waiting for real time — essential for testing edge cases like grace period expiry and auto-renewal cancellation.

Account creation API Device onboarding Mongo state control Grace/Expired/Active pri-qa + maint-beta DB change history
5
Stripe Preview

Multi-Country Pricing & Checkout Validation

Validates Stripe hosted checkout for 38+ countries across 1-year and 3-year prepaid plans. For each country, the runner creates a fresh mail.tm account, navigates to Stripe checkout, fills all required billing fields (address rules vary significantly by country), and verifies the displayed price matches the expected value.

Screenshots before and after fill, plus a pass/fail match indicator, are collected per country and preserved in timestamped run folders for post-release review.

⭐ CEO Live Demo Moment

Demonstrated live to
CJ Prober, CEO of NETGEAR

This dashboard was not built for the hackathon. It was already running in production, and it had already proven its value in the most direct way possible — by surfacing a real bug, live, in front of the company's CEO.

The Setup

QA was running a live demonstration of the automation dashboard to CJ Prober. The suite was the 2-TC demo: TC1 checking 3-year Stripe pricing, TC2 validating NHB device auto-allocation.

The Discovery

As TC1 executed in real-time, the dashboard populated its result: FAILED. The Stripe billing portal was showing $26.97 for the 3-year plan — but the Figma design specification cited $29.97 ($9.99/device/year × 3).

CJ's Reaction

The bug was discovered in the room, in real-time, while CJ watched the dashboard update live. He was impressed by the system's ability to surface discrepancies immediately — a complete closed loop: design spec → automated assertion → live result → instant visibility.

Resolution

The discrepancy was filed as PRJCBUGEN-60719 and investigated cross-team. Post-investigation confirmed: $26.97 is the correct 3-year prepay price ($8.99/device/year × 3 = $26.97, ~10% prepay discount). ✓ RESOLVED

Today

The Figma spec was updated. The 3-year pricing test now passes, verifying the confirmed correct pricing. The automation that found the bug proves it's fixed.

"Running in real-time, in a closed loop — from test trigger to live result to bug discovery."

— The moment CJ saw the dashboard surface a pricing discrepancy live

Demo note — Mongo access

QA Mongo controls (Move to Grace / Expired / Active) use an in-memory password for the selected env. On first use after starting the dashboard, the UI prompts once; the password lives only for the life of the running Node server process. Close the browser → password stays; stop node dashboard/server.js → password is wiped and the next run prompts again. No credentials are written to disk or committed.

Insight QA Workbench MCP

A stdio Model Context Protocol server ships in this repo at mcps/NTGR-Insight_QA/. It does not replace the dashboard — it is an HTTP client to this same running server (node dashboard/server.js), so agents can run tests, drive Account Navigator APIs, and fetch artifacts without clicking the UI for every step.

01

Give the README to your agent

Open mcps/NTGR-Insight_QA/README.md in this repo and add it as context (or paste it) into your preferred coding agent — Cursor, Claude Code, or similar. That file is the full install guide: dependencies, cursor-mcp.example.json, env vars, and the tool catalog for server NTGR-Insight_QA. Your agent can walk through it and wire the MCP for you (e.g. merge the example into .cursor/mcp.json, set args to index.js, run npm install under mcps/NTGR-Insight_QA/).

02

Next: ask how to use the MCP server

After the agent has registered NTGR-Insight_QA and reloaded MCP, ask it as a follow-up to explain how to use the server in practice — for example insight_qa_health, insight_qa_nav_account_defaults, running tests via insight_qa_tests_run, or Account Navigator flows. Keep node dashboard/server.js running so the MCP has a live dashboard to call.

Reproducibility

Setup Guide. Zero Ambiguity.

Every dependency is declarative. Every step is deterministic. From a bare machine — macOS, Ubuntu, or Windows — to a running dashboard in under five minutes. setup.sh detects your OS and installs Node.js 20 automatically via the right native package manager.

🍎 macOS 13+
🐧 Ubuntu 20.04+ / Debian
🪟 Windows 10 / 11
🟢 Node.js 20+ LTS (auto-installed)
🎭 Playwright 1.58+
🌐 Chromium (auto-installed)
📡 NETGEAR VPN / Network access
⚡ One-liner — run this in the project root
Download setup.sh
$ chmod +x setup.sh && ./setup.sh
01

Clone / unzip the repository

Obtain the project folder from GitHub or the submitted zip archive. All source files, the Playwright config, and dashboard server are included — no manual patching required.

git clone <repo> && cd playwright_hackathon
02

Install Node.js 20 LTS — OS-aware

setup.sh detects your OS and installs Node 20 via the right native package manager: Homebrew on macOS, apt / dnf / pacman on Ubuntu/Debian/Fedora/Arch (and WSL), winget / Chocolatey / nvm-windows on Windows. A companion setup.ps1 is available for PowerShell users who don't have Git Bash.

./setup.sh # or .\setup.ps1
03

Install npm dependencies

All runtime and dev dependencies — @playwright/test, ws, express, chokidar — are declared in package.json. The lockfile guarantees byte-identical installs across machines.

npm install
04

Install Playwright browser binaries

Playwright manages its own isolated Chromium build — no system Chrome dependency, no version mismatch risk. The binary is pinned to the exact version declared by @playwright/test.

npx playwright install chromium
05

Bootstrap Account Navigator tool

The Account Navigator runs as an embedded Node.js API server. Install its dependencies separately — they are isolated from the root project to prevent version conflicts with the Playwright toolchain.

cd tools/account-navigator && npm install && cd ../..
06

Provision authentication state

The test suite uses persistent browser context for session reuse. Run the bootstrap login once to populate .auth/ with a valid Chromium storage state. Subsequent test runs rehydrate from this snapshot, skipping the login wall.

mkdir -p .auth test-results/evidence
07

Launch the dashboard server

A single Node.js process serves the HTML dashboard, WebSocket event bus, test runner API, and static assets — all on port 9324. No separate web server, no reverse proxy.

node dashboard/server.js
08

Open dashboard & fire the suite

Navigate to localhost:9324, confirm the WebSocket indicator is green, then click ▶ Run Tests. The orchestrator spawns two Playwright workers simultaneously — one per environment — and streams results live.

open http://localhost:9324
🧩

Chrome launch uses stealth-style flags only — no CAPTCHA extension is loaded. Disposable-mail flows (e.g. Yopmail) may still show challenges; the suite waits for the page to become usable or falls back to mail.tm when Yopmail cannot complete.

⚡

Parallel execution model: the suite spawns 2 Playwright workers concurrently. Worker 1 runs 5 TCs against pri-qa (1-year account). Worker 2 runs 5 TCs against maint-beta (3-year account). Both streams appear in the dashboard simultaneously — 10 tests, 2 environments, one click.

✅

Cross-platform: the same setup.sh has been tested on macOS 13+, Ubuntu 22.04 / Debian 12, and Windows 10 / 11 (Git Bash + WSL2). It auto-detects your OS and routes Node installation through the right native package manager — Homebrew, apt, dnf, pacman, winget, Chocolatey, or nvm-windows. Windows users without Git Bash can run the PowerShell equivalent setup.ps1. The Playwright runner, dashboard server, WebSocket layer, and MCP are fully cross-platform.
